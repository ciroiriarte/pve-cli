# Contributing to pve-cli

Thanks for helping! This is an unofficial Proxmox VE/PDM client (see
[`NOTICE`](NOTICE)). It's a Go module; the command is `pc`.

## Build, test, lint

Requires Go ≥ 1.22.

```bash
make build    # ./pc
make test     # unit + httptest integration tests
make vet      # go vet
make docs     # man pages + completions + markdown reference -> dist/ (gitignored)
gofmt -l .    # must print nothing
```

CI gate: `go vet`, `gofmt`, and `go test ./...` must all be green.

## Project layout

```
cmd/pc          main entrypoint
cmd/gendocs     man/completion/markdown generator (make docs)
cmd/schemagen   ingest upstream apidoc.js -> committed schema snapshot; also -diff
internal/
  cli/          cobra command tree (the curated, stable UX)
  transport/    HTTP client: TLS (+fingerprint pin), retries, rate limit, auth injection
  auth/         token + ticket providers
  protocol/     Proxmox envelope/error decode, UPID/task types
  provider/     backend interface; provider/pve and provider/pdm implementations
  domain/       provider-neutral models (Guest, Node, Storage, Remote, …)
  config/       config file, profiles/contexts, precedence, secret refs
  output/       table/json/yaml/csv/value rendering
  schema/       API-schema IR: parse, resolve, diff, golden signature
  generated/    go:embed'd schema snapshots (pve, pdm) — regenerated by schemagen
  task/         UPID polling / wait / spinner
```

Design rules:
- The `cli` layer depends only on the `provider` interface + `domain` types +
  the output formatter — never on raw HTTP.
- Curated mutations are thin wrappers over `provider.Raw` + the shared
  `rawMutate` → `TaskHandle` helper, so they all get `--wait`/`--no-wait`/
  `--wait-timeout` and destructive confirmation for free.
- `json`/`yaml` output is the stable contract; table layout is not.

## Adding a curated command

1. Add the cobra command under `internal/cli/` (mirror an existing one, e.g.
   `guest_lifecycle.go`). Reuse `resolveGuest`, `rawMutate`, `finishTask`,
   `rawListTabular`, and `confirm`.
2. Register it in `internal/cli/root.go` (or the parent group).
3. Add an httptest-backed test (see `internal/cli/*_test.go` + the `runCLI`
   and `withCreds` helpers).
4. Keep destructive actions behind `confirm(...)` / `--yes`.

## Regenerating the API schema

The embedded snapshots drive `pc raw` and the golden tests.

```bash
go run ./cmd/schemagen -version 9 -out internal/generated/pve/apischema.json   # PVE (fetches apidoc.js; -in FILE for a local copy)
go run ./cmd/schemagen -version 0 -out internal/generated/pdm/apischema.json -url https://pdm.proxmox.com/docs/api-viewer/apidoc.js  # PDM
PVE_CLI_UPDATE_GOLDEN=1 go test ./internal/generated/...                       # refresh golden signatures
go run ./cmd/schemagen -diff old.json new.json                                 # report added/removed/changed
```

The golden tests fail on unreviewed schema drift — regenerate intentionally.

## Versioning & the v1.0.0 gate

[SemVer 2.0.0](https://semver.org). While on `0.x`, the CLI/config/`json`
surface may change between minors (breaking changes are noted in
[CHANGELOG.md](CHANGELOG.md)).

**Hard release gate:** `v1.0.0` must NOT be tagged until the full live test
battery (auth incl. ticket, list/show, create→poll→cleanup lifecycle, error
cases) passes against **at least one real instance of every supported endpoint
type — currently PVE *and* PDM**. Until then, releases stay on `0.x`.

## Packaging

Native packages build on build.opensuse.org (`home:ciriarte:pve-cli`); recipes
live in [`packaging/obs/`](packaging/obs/). Bump `packaging/obs/_service`
`revision` and `debian/changelog` on release.

## Commit / PR conventions

- Conventional-commit-ish prefixes (`feat:`, `fix:`, `build:`, `docs:`).
- Include tests for behavior changes; keep `gofmt`/`vet`/tests green.
- One logical change per PR; explain the "why".
